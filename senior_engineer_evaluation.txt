strengths:
  - id: S1
    title: "Declarative registration without metaclass magic"
    detail: >
      __init_subclass__ walks the MRO in reverse so subclasses inherit and
      override parent key bindings and hooks deterministically, and duplicate
      bindings within a class raise at class-definition time rather than at
      runtime. This is the right pattern, executed correctly and covered by
      tests (test_decorated_key_bindings_and_hooks_are_inherited,
      test_duplicate_local_key_bindings_raise_error).
  - id: S2
    title: "Defensive curses handling"
    detail: >
      addstr/center_text return booleans instead of raising, bounds-check
      against getmaxyx, truncate by display width, and tolerate curses.error —
      the most common crash class in curses apps (resize/narrow terminals) is
      designed out.
  - id: S3
    title: "Eager input validation with good error messages"
    detail: >
      TuiConfig, key_binding, and lifecycle_hook all fail fast with messages
      that name the offending value and list valid alternatives.
  - id: S4
    title: "Consistent modern Python style"
    detail: >
      Type hints throughout, `from __future__ import annotations`, slots
      dataclass, Literal-typed lifecycle stages, MappingProxyType for the
      public keymap view, ruff configured in pyproject.toml.
  - id: S5
    title: "Tutorial curriculum as living documentation"
    detail: >
      Five progressive examples each with a local README, plus a test
      (test_hello_world_preview.py) that pins the curriculum sequence so docs
      and code cannot silently diverge.

  - id: F6
    severity: medium
    category: performance
    title: "Event loop busy-spins when refresh_rate is 0 and adds input latency otherwise"
    location: "pyTuiMonster/app.py:180"
    detail: >
      With nodelay(True) and timeout(0), a refresh_rate of 0 produces a 100%
      CPU spin loop; with the default 1/30, keypresses wait up to 33 ms in
      time.sleep before being processed. The blocking sleep also couples frame
      rate to input latency.
    recommendation: >
      Use stdscr.timeout(int(refresh_rate * 1000)) and let getch() perform the
      frame pacing, or use curses.napms; document the refresh_rate=0 behavior.
  - id: F7
    severity: low
    category: code_quality
    title: "Duplicated display-width logic"
    location: "pyTuiMonster/app.py:18; pyTuiMonster/app.py:225"
    detail: >
      _display_width and _truncate_to_width re-implement the same
      combining-mark / east-asian-width classification; a change to one (e.g.
      adding zero-width-joiner handling) will silently miss the other.
    recommendation: "Extract a shared per-character width helper."
  - id: F8
    severity: low
    category: api_design
    title: "Inconsistent validation between decorator and runtime registration"
    location: "pyTuiMonster/app.py:107"
    detail: >
      @key_binding validates key types and duplicates, but
      register_key_handler/register_key_handlers accept anything and silently
      overwrite existing bindings, including config.stop_keys collisions
      (stop_keys win because they are checked first, which is undocumented).
    recommendation: >
      Validate runtime registrations the same way, and document handler
      precedence and the StopIteration-stops-the-app convention
      (pyTuiMonster/app.py:221), which is currently an undocumented surprise.
  - id: F9
    severity: low

    recommendation: >
      Consider renaming to `pytuimonster` (or `tui_monster`) before the API is
